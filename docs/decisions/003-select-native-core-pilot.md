# ADR 003: Select shared-core/native-view pilot

- **Status:** evidence recorded; no architecture selected
- **Date:** 2026-06-20
- **Decision evidence:** [`proof/select/comparison.json`](../../proof/select/comparison.json)

## Question

Can a framework-neutral Select model with native React, Vue, Svelte, and Solid views match the immutable TypeScript control, and does it compare favorably with the Mitosis Select sketch?

## Evidence boundary

Facts below are derived from receipts and source, not earlier narrative summaries. The immutable control is `@cloudflare/kumo@2.5.2`; its selected receipts are browser-backed across four frameworks. Its React provenance lists a 7,986-byte runtime chunk, a 138-byte entry, and 10,698 bytes of principal Select declarations. Those are package-file bytes, not bundle-cost estimates.

The shared model has **41 LOC / 10,052 source bytes**. Native views own **43 React, 79 Vue, 180 Svelte, and 69 Solid LOC** (7,045 / 6,031 / 7,124 / 7,267 bytes). Harness adapters are three fixture files per framework. Rendering, reactive wiring, events, focus/scroll and framework SSR/hydration remain native escape hatches; therefore shared model size is not total ownership.

Mitosis emits small sources (React 400, Vue 488, Svelte 336, Solid 517 bytes) from a native `<select>` sketch. Its receipt proves generation and provenance only. Public-behavior parity, builds, types, styles/package behavior, SSR and hydration are **not-run**. It also carries a Mitosis 0.13.2 generator API adapter. Generated bytes do not measure equivalent behavior.

## Results (keep the categories separate)

- **Contract:** the model vectors pass **8/8**. This proves reducer/model semantics only.
- **Structural:** package/type/export/style gates pass 4/4. Vue and Solid client builds and SSR pass. These are not browser behavior passes.
- **Observed browser failure:** Solid actually reaches the browser and fails pointer open/select, typeahead, controlled/uncontrolled value/open, and hydration. Keyboard, disabled-option, console/network and server-node checks pass; callback ordering, DOM/ARIA and focus/scroll remain blocked after the failed flow.
- **Blocked browser gates:** React fails before execution because its JSX resolves to `solid-js/jsx-runtime`; Svelte cannot load its internal client modules; Vue builds and SSRs, then its browser assertion dereferences a missing node. Their interaction/hydration gates are **blocked**, not failed behavior and not passed.
- **Aggregate shared-core receipt:** **24 passed, 10 failed, 22 blocked, 0 not-run**; verdict `failed`, fail-closed.
- **Consumer DX:** separate package-pilot receipts contain **10 passed / 21 blocked / 6 not-run**. These counts describe the current receipt; they do not establish install/build/hydration readiness.

## Maintenance implication

The control has immutable package and browser provenance but remains the full-emitter ownership baseline. Shared-core centralizes model changes, yet every behavior or upstream API change can require four native integrations plus harness work; current hydration/build failures show that boundary is material. Mitosis minimizes emitted source but currently externalizes unproved behavior and generator-upgrade risk. LOC and bytes cannot rank these unequal scopes.

## Pilot verdict

**No winner is declared.** The pilot establishes a useful model contract and exposes product/evidence gaps; it does not establish parity or readiness.

The next milestone is to repair Select: make all four target builds deterministic, fix Solid hydration/interaction, complete public API/event-order/ARIA/focus/style/package checks, and rerun immutable receipts. Dialog should follow only after that, or proceed strictly as a bounded contract experiment—not as a parity campaign.
