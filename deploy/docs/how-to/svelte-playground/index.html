How-to guide

Svelte playground validation

We tested the real package-name import in svelte.dev Playground. The component source is valid, but the playground cannot resolve the package until it exists in the public npm registry.

Open the exact attempted import ↗Run the tarball-backed example

What was tested

Observed result

The browser run reached Svelte 5.56.3, loaded the encoded source, and then failed during dependency resolution. The package name returned HTTP 404 from both the public npm registry and jsDelivr’s npm resolver.

Why a tarball URL cannot fix this inside svelte.dev

The playground’s worker parses bare imports, asks jsDelivr for a public npm version, then fetches the corresponding tarball from registry.npmjs.org. Its source format has component files and aliases, but no package dependency map for arbitrary remote tarball URLs.

Classification: registry-publication blocked, not component blocked. The same production tarball passes npm installation, type checking, client build, SSR, hydration, browser behavior, CSS, tree-shaking, and workspace checks.

What works now

  • The complete Svelte example installs the production HTTPS tarball.
  • The live gallery is compiled from that installed tarball.
  • The downloadable project runs locally and in dependency-capable environments that accept URL dependencies.

What unlocks direct svelte.dev import

Publish @acoyfellow/kumo-svelte@0.0.1 to the public npm registry under an authorized Cloudflare publishing identity. Publication is intentionally not simulated and is not performed without registry ownership and release approval.
