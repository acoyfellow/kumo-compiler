Distribution and playgrounds

A package can be installable without being discoverable by every package resolver. Kumo’s current artifacts are real npm tarballs delivered over HTTPS, but they are not public registry entries.

What works

  • npm install https://…/kumo-svelte-0.0.1.tgz
  • npm lockfiles and workspaces
  • Vite client and SSR builds
  • Dependency-capable cloud sandboxes that accept URL dependencies
  • Kumo’s package-backed production galleries

What does not work yet

Resolvers that only accept a public npm package name cannot find @cloudflare/kumo-svelte, @cloudflare/kumo-vue, or @cloudflare/kumo-solid. The svelte.dev Playground is one such resolver.

Why not fake it

Copying component source into a playground would show a lookalike, not package installation. Rewriting imports to a CDN bundle would prove a different distribution artifact. The playground receipt therefore records the actual package-name failure and points to the tarball-backed example for executable proof.

What registry publication changes

Publishing the same package names to the public npm registry would let svelte.dev resolve the import through jsDelivr and registry tarballs. That is an ownership and release-policy action, not a compiler change. It requires an authorized Cloudflare publishing identity.

Current status: package implementation passed; hosted distribution passed; public-registry discovery blocked.
